      Steps
      Detallar el flujo de eventos del caso de uso

      Ya debería tener una breve descripción paso a paso del flujo de eventos del caso de uso. Esto se crea en la Actividad: Buscar actores y casos de uso . Utilice esta descripción paso a paso como punto de partida y gradualmente hágala más detallada.

      Describir los casos de uso de acuerdo con los estándares decididos para el proyecto (documentados en el Artefacto: Pautas de modelado de casos de uso ). Decida los siguientes puntos antes de describir los casos de uso para que sea coherente entre ellos:

      ¿Cómo comienza el caso de uso? El inicio del caso de uso debe describir claramente la señal que activa el caso de uso. Escriba, por ejemplo, "El caso de uso puede comenzar cuando... suceda".
      ¿Cómo termina el caso de uso? Debe indicar claramente lo que sucede en el curso del flujo para finalizar el caso de uso. Escriba, por ejemplo, "Cuando... sucede, el caso de uso finaliza".
      ¿Cómo interactúa el caso de uso con los actores? Para minimizar cualquier riesgo de malentendidos, diga exactamente qué residirá dentro del sistema y qué residirá fuera del sistema. Estructura la descripción como una serie de párrafos, en los que cada párrafo expresa una acción en el formato: "Cuando el actor hace..., el sistema hace...". También puede enfatizar la interacción escribiendo que el caso de uso envía y recibe señales de los actores, por ejemplo: "El caso de uso comienza cuando recibe la señal 'inicio' del Operador".
      ¿Cómo intercambia datos el caso de uso con un actor? Si lo desea, puede consultar los argumentos de las señales, pero sería mejor escribir, por ejemplo, "El caso de uso comienza cuando el Usuario inicia sesión en el sistema dando su nombre y contraseña".
      ¿Cómo repite el caso de uso algún comportamiento? Deberías intentar expresar esto en lenguaje natural. Sin embargo, en casos excepcionales, podría valer la pena utilizar construcciones similares a códigos, como "WHILE-END WHILE", "IF-THEN-ELSE" y "LOOP-END LOOP", si los términos correspondientes en lenguaje natural son difíciles. para expresar. Sin embargo, en general, debe evitar el uso de construcciones tipo código en las descripciones de casos de uso porque son difíciles de leer y mantener.
      ¿Existen situaciones opcionales en el flujo de eventos de un caso de uso? A veces a un actor se le presentan varias opciones. Los miembros del equipo deben escribir esto de la misma manera. Por ejemplo:
      "El actor elige una de las siguientes, una o más veces:

      a) . . .

      b) . . .

      C) . . ".

      ¿Cómo se debe describir el caso de uso para que el cliente y los usuarios puedan entenderlo? El uso de terminología específica de la metodología, como caso de uso, actor y señal, puede hacer que el texto sea innecesariamente difícil de entender. Para que el texto sea más fácil de leer, puedes enumerar las acciones o adoptar alguna otra estrategia. Cualquier estrategia que utilice debe especificarse en las pautas generales de modelado de casos de uso para que la tenga en cuenta durante toda la actividad de descripción de casos de uso.
      Concéntrese en describir lo que se hace en el caso de uso, no cómo se deben resolver los problemas específicos internos del sistema. Cuando trabaje con modelos de objetos, es posible que deba complementar la descripción con detalles sobre cómo funcionan las cosas, así que no haga la descripción demasiado detallada en este punto. Describe sólo lo que crees que será estable más adelante.

      Si el flujo de eventos de un caso de uso se ha vuelto demasiado amplio o complejo, o si parece tener partes independientes entre sí, divídalo en dos o más casos de uso.

      Cuando escribas el texto descriptivo, consulta el glosario. A medida que surjan nuevos términos a partir de nuevos conceptos, inclúyalos en el glosario. No cambie la definición de un término sin informar a los miembros apropiados del proyecto.

      El contenido de una descripción del flujo de eventos
      Una descripción del flujo de eventos explora:

      Cómo y cuándo comienza el caso de uso.
      Ejemplo:
      "El caso de uso puede comenzar cuando un usuario activa la función 'Administrar pedido'".

      Cuándo el caso de uso interactúa con los actores y qué datos intercambian.
      Ejemplo:
      "Para crear un nuevo pedido, el usuario activa la función 'Nuevo' y luego especifica los siguientes datos obligatorios sobre el pedido: nombre, elementos de red (al menos uno) y tipo de función de medición. También se pueden especificar datos opcionales sobre el pedido: un comentario (una pequeña descripción textual). Luego el usuario activa la función 'Ok' y se crea un nuevo pedido en el sistema."

      Nota: Debes ser explícito respecto de los datos intercambiados entre los actores y el caso de uso; de lo contrario, el cliente y los usuarios probablemente no entenderán la descripción del caso de uso.

      Cómo y cuándo el caso de uso utiliza datos almacenados en el sistema o almacena datos en el sistema.
      Ejemplo:
      "El usuario activa la función 'Modificar' para modificar un pedido existente y especifica un número de pedido (entero pequeño). Luego, el sistema inicializa un formulario de pedido con el nombre del pedido, sus elementos de red y su tipo de función de medición. Estos datos se recuperan de un dispositivo de almacenamiento secundario."

      Cómo y cuándo finaliza el caso de uso.
      Ejemplo:
      "El caso de uso finaliza cuando el Solicitante activa la función 'Salir'."

      También debe describir flujos de eventos extraños o excepcionales. Un flujo excepcional es un subflujo del caso de uso que no se adhiere al comportamiento normal o básico del caso de uso. No obstante, este flujo puede ser necesario en cualquier descripción completa del comportamiento del caso de uso. Un ejemplo típico de flujo excepcional es el que se da en el primer ejemplo. Si el caso de uso recibe algunos datos inesperados (que el actor no es el esperado en ese contexto particular), finaliza. Tener al actor equivocado y terminar prematuramente no está en el flujo típico de eventos.

      Otras cosas que se deben y no se deben considerar al describir un caso de uso incluyen:

      Describa el flujo de eventos, no solo la funcionalidad o el propósito del caso de uso.
      Describa sólo los flujos que pertenecen al caso de uso, no lo que sucede en otros casos de uso que funcionan en paralelo con él.
      No menciones actores que no se comunican con el caso de uso en cuestión.
      No proporcione demasiados detalles cuando describa la interacción del caso de uso con cualquier actor.
      Si el orden de los subflujos descritos para el caso de uso no tiene que arreglarse, no lo describa como si tuviera que arreglarlo.
      Utilice los términos del glosario común y considere lo siguiente al escribir el texto:
      Utilice vocabulario sencillo. No utilice un término complejo cuando uno simple sea suficiente.
      Escribe oraciones cortas y concisas.
      Evite los adverbios como muy, más, más bien y similares.
      Utilice puntuación correcta.
      Evite las oraciones compuestas.
      Para obtener más información, consulte Directrices: Caso de uso , las discusiones sobre contenidos y estilo del flujo de eventos.

      Estructurar el flujo de eventos del caso de uso

      El flujo de eventos de un caso de uso se puede dividir en varios subflujos. Cuando se activa el caso de uso, los subflujos se pueden combinar de varias maneras si se cumple lo siguiente:

      El caso de uso puede partir de uno de varios caminos posibles, dependiendo de la entrada de un actor determinado o de los valores de algún atributo u objeto. Por ejemplo, un actor puede decidir, entre varias opciones, qué hacer a continuación, o el flujo de eventos puede diferir si un valor es menor o mayor que cierto valor.
      Ejemplo:
      Parte de la descripción del caso de uso Retirar dinero en un sistema de cajero automático podría ser "La cantidad de dinero que el cliente desea retirar de la cuenta se compara con el saldo de la cuenta. Si la cantidad de dinero excede el saldo, el Se informa al cliente y el caso de uso finaliza. De lo contrario, el dinero se retira de la cuenta".

      El caso de uso puede realizar algunos subflujos en secuencias opcionales.
      El caso de uso puede realizar varios subflujos al mismo tiempo.
      Debe describir todos estos flujos opcionales o alternativos. Se recomienda describir cada subflujo en un suplemento separado de la sección Flujo de eventos, y debería ser obligatorio para los siguientes casos:

      Subflujos que ocupan un gran segmento de un flujo de eventos determinado.
      Flujos excepcionales de eventos. Esto ayuda a que el flujo básico de eventos del caso de uso se destaque más claramente.
      Cualquier subflujo que se pueda ejecutar en varios intervalos en un mismo flujo de eventos.
      Si un subflujo involucra sólo una parte menor del flujo completo de eventos, es mejor describirlo en el cuerpo del texto.

      Ejemplo:
      "Este caso de uso se activa cuando cualquiera de los actores Ordenante o Administrador de Performance Manager solicita la función 'administrar orden'. Si la señal no proviene de uno de estos actores, el caso de uso finalizará la operación y mostrará un mensaje apropiado. mensaje al usuario Sin embargo, si se reconoce al actor, el caso de uso continúa por...".

      Puede ilustrar la estructura del flujo de eventos con un diagrama de actividades; consulte Directrices: Diagrama de actividades en el modelo de casos de uso .

      Para obtener más información, consulte Pautas: Caso de uso, estructura del flujo de eventos.

      Ilustrar las relaciones con los actores y otros casos de uso

      Cree diagramas de casos de uso que muestren el caso de uso y sus relaciones con los actores y otros casos de uso. Un diagrama de este tipo funciona como un diagrama local del caso de uso y debe estar relacionado con él. Tenga en cuenta que este tipo de diagrama de casos de uso local suele tener poco valor, a menos que el caso de uso tenga relaciones entre casos de uso que deban explicarse, o si existe una complejidad inusual entre los actores involucrados.

      Para obtener más información, consulte Directrices: Diagrama de casos de uso.

      Describir los requisitos especiales del caso de uso

      Cualquier requisito que pueda estar relacionado con el caso de uso, pero que no se tenga en cuenta en el Flujo de Eventos del caso de uso, debe describirse en los Requisitos Especiales del caso de uso. Es probable que tales requisitos no sean funcionales.

      Para obtener más información, consulte Directrices: Caso de uso, requisitos especiales.

      Describir protocolos de comunicación
      Desarrollar un protocolo de comunicación si el actor es otro sistema o hardware externo. La descripción del caso de uso debe indicar si se va a utilizar algún protocolo existente (tal vez incluso uno estandarizado). Si el protocolo es nuevo, debe describirlo completamente durante el desarrollo del modelo de objetos.
      Describir las condiciones previas del caso de uso

      Una condición previa de un caso de uso explica el estado que debe tener el sistema para que sea posible iniciar el caso de uso.

      Ejemplo:
      Para que un sistema de cajero automático pueda dispensar efectivo, se deben cumplir las siguientes condiciones previas:

      La red de cajeros automáticos debe ser accesible.
      El cajero automático debe estar en condiciones de aceptar transacciones.
      El cajero automático debe tener al menos algo de efectivo a mano que pueda dispensar.
      El cajero automático debe tener suficiente papel para imprimir un recibo de al menos una transacción.
      Todas estas serían condiciones previas válidas para el caso de uso Dispense Cash.

      Tenga cuidado de describir el estado del sistema; Evite describir los detalles de otras actividades incidentales que puedan haber tenido lugar antes de este caso de uso.

      Las condiciones previas no se utilizan para crear una secuencia de casos de uso. Nunca habrá un caso en el que primero tenga que realizar un caso de uso y luego otro, para tener un flujo de eventos significativo. Si cree que es necesario hacer esto, es probable que haya descompuesto demasiado el modelo de caso de uso. Corrija este problema combinando los casos de uso secuencialmente dependientes en un único caso de uso. Si esto hace que el caso de uso resultante sea demasiado complejo, considere técnicas para estructurar casos de uso, como se presenta en Estructurar el flujo de eventos del caso de uso anterior, o en la Actividad: Estructurar el modelo de caso de uso .

      Para obtener más información, consulte Directrices: caso de uso, condiciones previas y condiciones posteriores.

      Describir las condiciones posteriores del caso de uso

      Una poscondición en un caso de uso enumera los posibles estados en los que puede encontrarse el sistema al final del caso de uso. El sistema debe estar en uno de esos estados al final de la ejecución del caso de uso. También se utiliza para indicar acciones que realiza el sistema al final del caso de uso, independientemente de lo que ocurrió en el caso de uso.

      Ejemplo :
      Si el cajero automático siempre muestra el mensaje de "Bienvenida" al final de un caso de uso, esto podría documentarse en la condición posterior del caso de uso.

      De manera similar, si el cajero automático siempre cierra la transacción del cliente al final de un caso de uso como Retirar efectivo, independientemente del curso de los acontecimientos, ese hecho debe registrarse como una condición posterior para el caso de uso.

      Las poscondiciones se utilizan para reducir la complejidad y mejorar la legibilidad del flujo de eventos del caso de uso.

      Bajo ninguna circunstancia se deben utilizar condiciones posteriores para crear una secuencia de casos de uso. Nunca debería haber un caso en el que primero deba realizar un caso de uso y luego otro, para tener un flujo de eventos significativo. Si siente la necesidad de hacer esto, los casos de uso secuencialmente dependientes deben combinarse en un solo caso de uso. Si esto hace que el caso de uso combinado sea demasiado complejo, considere técnicas para estructurar casos de uso, como se presenta en Estructurar el flujo de eventos del caso de uso arriba, o en la Actividad: Estructurar el modelo de caso de uso .

      Para obtener más información, consulte Directrices: caso de uso, condiciones previas y condiciones posteriores.

      Describir puntos de extensión
      Si el caso de uso se va a ampliar con otro caso de uso (consulte Directrices: Ampliar relación ), debe describir cuáles son los puntos de extensión (consulte Directrices: Caso de uso, puntos de extensión).
      Evalúe sus resultados

      Revisar y discutir el caso de uso con las partes interesadas, para que tengan una comprensión clara del caso de uso y estén de acuerdo con su descripción.

      La descripción del caso de uso está completa sólo cuando describe todo lo que el caso de uso realiza, implementa o permite de otro modo, de principio a fin. Antes de terminar, verifique que el caso de uso exhiba las propiedades que lo caracterizan como un "buen" caso de uso. Consulte los puntos de control para casos de uso e informes de casos de uso en Actividad: Revisar requisitos.
