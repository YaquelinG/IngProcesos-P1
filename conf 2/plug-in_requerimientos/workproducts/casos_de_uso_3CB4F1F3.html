Artifact: Casos de uso
Un caso de uso define un conjunto de instancias de casos de uso, donde cada instancia es una secuencia de acciones que realiza un sistema y que produce un resultado observable de valor para un actor en particular.
Available States: descrito, esquemas
Purpose

Las siguientes personas utilizarán los casos de uso:

Los clientes utilizarán los casos de uso para comprender el comportamiento del sistema y, dado que deben aprobar el flujo de eventos del caso de uso, el cliente también utilizará los casos de uso para aprobar el resultado del modelado de casos de uso.
Los usuarios potenciales utilizarán el caso de uso para comprender el comportamiento del sistema.
Los arquitectos de software utilizarán los casos de uso para identificar funcionalidades arquitectónicamente significativas.
Las personas que analizan , diseñan e implementan el sistema utilizarán el caso de uso para comprender el comportamiento requerido del sistema y perfeccionarlo.
Los diseñadores de casos de uso utilizarán los flujos de eventos de los casos de uso para encontrar clases. (Estos son los artefactos más importantes para los diseñadores de casos de uso).
Los evaluadores utilizarán los casos de uso como base para identificar casos de prueba.
Los gerentes utilizarán los casos de uso para planificar y dar seguimiento al modelado de casos de uso.
Los redactores de documentación utilizarán los casos de uso para comprender qué secuencia de uso debe describirse en la documentación (como la guía del usuario del sistema).

Description
Main Description

La plantilla proporcionada para una especificación de caso de uso contiene las propiedades textuales del caso de uso. Este documento se utiliza con una herramienta de gestión de requisitos, como Rational RequisitePro, para especificar y marcar los requisitos dentro de las propiedades del caso de uso. 

Los diagramas del caso de uso se pueden desarrollar en una herramienta de modelado visual, como Rational Rose. Se puede generar un informe de caso de uso (con todas las propiedades) con Rational SoDA. 

Para obtener más información, consulte mentores de herramientas: Gestión de casos de uso utilizando Rational Rose y Rational RequisitePro y Creación de un informe de caso de uso utilizando Rational SoDA .

Momento
Los casos de uso se identifican y posiblemente se describen brevemente al principio de la fase inicial, para ayudar a definir el alcance del sistema. Los casos de uso que son relevantes para el análisis o el diseño arquitectónico del sistema se describen en detalle dentro de la fase de Elaboración. Los casos de uso restantes se describen en detalle en la fase de Construcción.

Responsabilidad
Un especificador de requisitos es responsable de la integridad del caso de uso, lo que garantiza que:

el caso de uso cumple con sus requisitos (es decir, describe correctamente la funcionalidad que es relevante para el caso de uso, y solo esta funcionalidad)
el flujo de eventos es legible y se adapta a su propósito
Las relaciones de casos de uso que se originan en el caso de uso están justificadas y se mantienen consistentes.
El papel del caso de uso en el que interviene en comunicaciones-asociaciones es claro e intuitivo.
Los diagramas que describen el caso de uso y sus relaciones son legibles y se adaptan a su propósito.
Los requisitos especiales son legibles y se adaptan a su propósito.
las condiciones previas son legibles y se adaptan a su propósito
las condiciones posteriores son legibles y se adaptan a su propósito
Se recomienda que el especificador de requisitos responsable de un caso de uso también sea responsable del paquete de casos de uso que lo adjunta. Para obtener más información, consulte Directrices: Paquete de casos de uso .

Sastrería
Decidir en qué medida se elaborarán los Casos de Uso:

¿Describir sólo los flujos principales?
¿Describir sólo los casos de uso más importantes?
describir completamente las condiciones previas y posteriores?
Algunos proyectos aplican casos de uso de manera informal para descubrir requisitos, pero documentan y mantienen estos requisitos de otra forma. La forma de adaptar los casos de uso puede depender del tamaño del proyecto, la experiencia, el conjunto de herramientas, la relación con el cliente, etc. Consulte Directrices: Caso de uso para obtener orientación relacionada con la adaptación de casos de uso.
