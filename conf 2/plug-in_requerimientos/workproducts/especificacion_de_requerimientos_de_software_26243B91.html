Artifact: Especificación de requerimientos de software
La Especificación de requisitos de software (SRS) captura los requisitos de software completos para el sistema o una parte del sistema. Cuando se utiliza el modelado de casos de uso, este artefacto consta de un paquete que contiene casos de uso del modelo de casos de uso y las Especificaciones complementarias aplicables.
Purpose

La Especificación de requisitos de software (SRS) se centra en la recopilación y organización de todos los requisitos que rodean su proyecto. Consulte el Plan de Gestión de Requisitos para determinar la ubicación y organización correcta de los requisitos. Por ejemplo, es posible que desee tener un SRS separado para describir los requisitos de software completos para cada característica en una versión particular del producto. Esto puede incluir varios casos de uso del modelo de casos de uso del sistema , para describir los requisitos funcionales de esta característica, junto con el conjunto relevante de requisitos detallados en Especificaciones complementarias .

La Especificación de requisitos de software se utiliza para recopilar los requisitos de software de su proyecto en un documento formal tipo IEEE 830, representado por una construcción de "paquete" UML.   Se proporcionan dos plantillas SRS de muestra:   una para usar con modelado de casos de uso (rup_srsuc.dot) y otra para usar sin modelado de casos de uso (rup_srs.dot). El primero (rup_srsuc.dot) hace referencia, o incluye, los artefactos del modelo de caso de uso: la encuesta del modelo de caso de uso , los informes de caso de uso y las especificaciones complementarias . Esto le permite tener un SRS formal compatible con IEEE sin tener que duplicar la información en estos otros 3 artefactos.    

El segundo (rup_srs.dot) es un documento independiente que contiene todos los requisitos de software directamente en el documento. Este documento requeriría que usted use la trazabilidad de los requisitos de artefactos del caso de uso, si se usan. Técnicamente, ambos contienen la misma información; sin embargo, la información en el modelo de casos de uso está encerrada por referencia (en lugar de duplicada) en el primero y completamente duplicada (si se usan casos de uso) en el segundo, lo que requeriría mucho más esfuerzo. en el mantenimiento de las relaciones de trazabilidad.  

Dado que es posible que se encuentre con varias herramientas diferentes para recopilar estos requisitos, es importante darse cuenta de que la colección de requisitos se puede encontrar en varios artefactos y herramientas diferentes. Por ejemplo, puede resultarle apropiado recopilar requisitos textuales, como requisitos no funcionales, restricciones de diseño, etc., con una herramienta de documentación de texto en Especificaciones complementarias . Por otro lado, puede resultarle útil recopilar algunos (o todos) los requisitos funcionales en los casos de uso y puede resultarle útil utilizar una herramienta adecuada a las necesidades de definir el modelo de caso de uso . Por este motivo, recopilaremos los requisitos para nuestro SRS en un paquete que puede ser un documento único o una colección de varios artefactos que describen los requisitos. (Ver: Directrices: Especificación de requisitos de software )

El paquete SRS controla la evolución del sistema durante toda la fase de desarrollo del proyecto, a medida que se agregan o modifican nuevas características al documento de Visión, se elaboran dentro del Paquete SRS. Las siguientes personas utilizan la Especificación de requisitos de software:

El analista de sistemas crea y mantiene la Visión y las Especificaciones Suplementarias , que sirven como entrada para el SRS y son el medio de comunicación entre el analista de sistemas, el cliente y otros desarrolladores.
El especificador de requisitos crea y mantiene el caso de uso individual y otros componentes del paquete SRS.
Los diseñadores utilizan el paquete SRS como referencia al definir responsabilidades, operaciones y atributos en las clases, y al ajustar las clases al entorno de implementación.
Los implementadores consultan el paquete SRS para obtener información al implementar clases.
El director del proyecto consulta el paquete SRS para obtener información al planificar iteraciones.
Los evaluadores utilizan el paquete SRS para verificar el cumplimiento del sistema.

Description
Main Description

La Especificación de requisitos de software (SRS) captura los requisitos de software completos para el sistema o una parte del sistema. A continuación se muestra un esquema SRS típico para un proyecto que utiliza modelado de casos de uso . Este artefacto consta de un paquete que contiene casos de uso del modelo de casos de uso y especificaciones complementarias aplicables y otra información de respaldo. Para obtener una plantilla de un SRS que no utiliza el modelado de casos de uso, que captura todos los requisitos en un solo documento, con las secciones aplicables insertadas de las Especificaciones Suplementarias (que ya no serían necesarias), consulte la plantilla HTML para un SRS sin casos de uso. .

Son posibles muchas disposiciones diferentes de un SRS. Consulte [ IEEE830-1998 ] para obtener más información sobre estas explicaciones, así como otras opciones para la organización SRS.

Momento
Las Especificaciones de Requisitos de Software van de la mano con los casos de uso y las Especificaciones Suplementarias , lo que implica que:

Inicialmente se consideran en la fase de inicio , como complemento a la definición del alcance del sistema.
Se perfeccionan de forma incremental durante las fases de elaboración y construcción .
Responsabilidad
Un especificador de requisitos es responsable de producir el paquete de especificación de requisitos de software (SRS), que es un complemento importante del modelo de casos de uso. El paquete SRS recopila especificaciones complementarias aplicables y casos de uso del modelo de casos de uso que, en conjunto, capturan un conjunto completo de requisitos en el sistema o un subsistema definido.

Sastrería
Son posibles muchas disposiciones diferentes de un SRS. Consulte [IEEE93] para obtener más información sobre estas explicaciones, así como otras opciones para la organización del SRS.
