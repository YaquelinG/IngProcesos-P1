Analizar el problema
El propósito de este detalle del flujo de trabajo es:
Llegar a un acuerdo sobre el problema que se está resolviendo,
Identificar las partes interesadas,
Definir los límites del sistema, y
Identificar las restricciones impuestas al sistema.

Main Description

El primer paso en cualquier análisis de problema es asegurarse de que todas las partes involucradas estén de acuerdo sobre cuál es el problema que estamos tratando de resolver con nuestro sistema. Para evitar malentendidos, es importante acordar una terminología común que se utilizará durante todo el proyecto. Desde el principio, debemos comenzar a definir los términos de nuestro proyecto en un glosario que se mantendrá durante todo el ciclo de vida del proyecto.

Para comprender completamente el problema que debemos abordar, es muy importante saber quiénes son nuestras partes interesadas . Tenga en cuenta que algunas de estas partes interesadas (los usuarios del sistema) estarán representadas por actores en nuestro modelo de casos de uso .

El Plan de Gestión de Requisitos proporcionará orientación sobre los artefactos de requisitos que se deben desarrollar, los tipos de requisitos que se deben gestionar para el proyecto, los atributos de requisitos que se deben recopilar y la trazabilidad de los requisitos que se utilizará en la gestión de los requisitos del producto.

El artefacto principal en el que documentamos la información del análisis del problema es el documento de Visión , que identifica la visión de alto nivel del usuario o cliente del sistema que se va a construir. En la Visión, los requisitos iniciales de alto nivel identifican las características clave que se desea que proporcione la solución adecuada. Por lo general, se expresan como un conjunto de características de alto nivel que el sistema podría poseer para resolver los problemas más críticos.

Las partes interesadas clave deben participar en la recopilación del conjunto de características a considerar, que podrían reunirse en un taller de requisitos . A las características se les deben asignar atributos como justificación, valor relativo o prioridad, fuente de solicitud, etc., para que las dependencias puedan comenzar a gestionarse.

Para determinar el alcance inicial de nuestro proyecto, se deben acordar los límites del sistema. El analista de sistemas identifica usuarios y sistemas (representados por actores ) que interactuarán con el sistema.

Si ha desarrollado un modelo de dominio , un modelo de caso de uso empresarial o un modelo de objeto empresarial , estos serán datos clave, junto con las reglas empresariales , para ayudar a realizar este análisis. Consulte también Directrices: Pasar de modelos de negocio a sistemas para obtener más orientación.

Este detalle del flujo de trabajo debe revisarse varias veces durante el inicio y la elaboración temprana . Luego, a lo largo del ciclo de vida del proyecto, se debe revisar según sea necesario mientras se gestionan los cambios inevitables que ocurrirán en nuestro proyecto, para garantizar que continuamos abordando los problemas correctos.
