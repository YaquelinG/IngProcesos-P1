Task: Detallar requerimientos de software
Purpose

Recopilar, detallar y organizar el conjunto (paquete) de artefactos que describen completamente los requisitos de software del sistema o subsistema

      Steps
      Detallar los requisitos del software

      Asegúrese de que todos los requisitos se especifiquen con el nivel de detalle necesario para entregárselos a los diseñadores, evaluadores y redactores de documentación. Revise los Puntos de control: Especificaciones complementarias para ver si se necesitan más detalles para capturar cualquier requisito de software no incluido en los casos de uso .

      Si elabora una Especificación de requisitos de software (SRS) formal , revise los Puntos de control: Especificación de requisitos de software .

      Si se rastrean o gestionan formalmente los requisitos, asegúrese de que cada requisito esté claramente identificado y etiquetado.

      Generar informes de soporte

      Los requisitos suelen almacenarse y gestionarse mediante una o más herramientas. Por ejemplo, herramientas para:

      modelado gráfico de casos de uso
      trazabilidad y gestión de requisitos
      otra documentación textual y gráfica.
      Este paso genera documentación a partir de estas herramientas para que la información pueda revisarse fácilmente.

      Cuando se utilizan herramientas de Rational, se aplican los siguientes informes:

      Encuesta sobre modelos de casos de uso ; y
      Informes de casos de uso.
      Si no se utilizan herramientas especializadas para capturar los requisitos, entonces este paso no es aplicable (todos los requisitos de software se escribirían directamente en la documentación).

      Empaquetar los requisitos para la revisión

      Para proyectos menos formales, este paso consiste en agrupar los informes relevantes y la documentación generada manualmente, con suficiente material de respaldo para que los requisitos puedan revisarse de manera efectiva.

      En proyectos más formales, una o más Especificaciones de requisitos de software (SRS) recopilan y organizan todos los requisitos relacionados con el proyecto. Por ejemplo, un SRS separado puede describir los requisitos de software completos para cada característica en una versión particular del producto. Esto puede incluir varios casos de uso del modelo de casos de uso del sistema , para describir los requisitos funcionales de esta característica, junto con el conjunto relevante de requisitos detallados en Especificaciones complementarias . Consulte el Plan de gestión de requisitos (parte del Plan de desarrollo de software ) para determinar la ubicación y organización correcta de los requisitos.

      La Especificación de requisitos de software es un documento formal de tipo IEEE 830, representado por una construcción de “paquete” UML.   Se proporcionan dos plantillas SRS de muestra:   una para usar con modelado de casos de uso (rup_srsuc.dot) y otra para usar sin modelado de casos de uso (rup_srs.dot). El primero (rup_srsuc.dot) hace referencia, o incluye, los artefactos del modelo de caso de uso: la encuesta del modelo de caso de uso , los informes de caso de uso y las especificaciones complementarias . Esto le permite tener un SRS formal compatible con IEEE sin tener que duplicar la información en estos otros 3 artefactos.    

      El segundo (rup_srs.dot) es un documento independiente que contiene todos los requisitos de software directamente en el documento. Este documento requeriría que usted use la trazabilidad de los requisitos de artefactos del caso de uso, si se usan. Técnicamente, ambos contienen la misma información; sin embargo, la información en el modelo de casos de uso está encerrada por referencia (en lugar de duplicada) en el primero y completamente duplicada (si se usan casos de uso) en el segundo, lo que requeriría mucho más esfuerzo. en el mantenimiento de las relaciones de trazabilidad.  

      Utilizando la plantilla de Especificación de requisitos de software , ensamble las piezas del paquete SRS y proporcione la información restante para tener una definición completa de los requisitos de software para este subsistema o característica.
