Task: Estructurar modelo de casos de uso
Purpose

- Extraer comportamiento en casos de uso que deben considerarse casos de uso abstractos. Ejemplos de tal comportamiento son el comportamiento común, el comportamiento opcional, el comportamiento excepcional y el comportamiento que se desarrollará en iteraciones posteriores.

- Encontrar nuevos actores abstractos que definan roles compartidos por varios actores.

      Steps
      Establecer relaciones de inclusión entre casos de uso

      Si un caso de uso contiene un segmento de comportamiento del cual solo el resultado, no el método para obtenerlo, es de importancia para el resto del caso de uso, este comportamiento se puede factorizar en un nuevo caso de uso de inclusión . El caso de uso original se convierte entonces en el caso de uso base en una relación de inclusión con el caso de uso de inclusión. Consulte también Directrices: modelo de caso de uso y Directrices: relación de inclusión .

      Una relación de inclusión entre dos casos de uso significa que una instancia de caso de uso que sigue a la descripción del caso de uso base también debe seguir la descripción del caso de uso de inclusión para estar completa.

      La relación de inclusión puede ayudar a aclarar un caso de uso al:

      Aislar y encapsular detalles complejos para que no oscurezcan el significado real del caso de uso.
      Mejorar la coherencia al incluir comportamientos que se incluyen en varios casos de uso básicos.
      Generalmente, más de un caso de uso debe incluir un caso de uso de inclusión para que valga la pena mantener un caso de uso adicional y la relación de inclusión.

      Sólo el caso de uso base conoce la relación entre los dos casos de uso; ningún caso de uso de inclusión sabe qué otros casos de uso lo incluyen.

      Describa la relación de inclusión indicando brevemente el propósito de la inclusión, así como la ubicación en el caso de uso base en el que se insertará la inclusión.

      Al describir el flujo de eventos del caso de uso base, debe hacer referencia a la inclusión en la ubicación en la que se inserta.

      Establecer relaciones extendidas entre casos de uso

      Si un caso de uso tiene segmentos de comportamiento que son de carácter opcional o excepcional y que no contribuyen a la comprensión del propósito principal del caso de uso, extráigalos a un nuevo caso de uso de extensión . El caso de uso original se convierte entonces en un caso de uso base , con el cual el caso de uso de extensión tiene una relación de extensión. Consulte también Directrices: modelo de caso de uso y Directrices: relación extendida .

      En el caso de uso base, usted declara puntos de extensión, que definen en qué parte del caso de uso base se pueden realizar extensiones. Consulte también Directrices: Caso de uso .

      Los subflujos complejos y el comportamiento opcional son los primeros candidatos a dividirse en casos de uso de extensión. A menudo, este comportamiento puede ser bastante complejo y difícil de describir: incluirlo en el flujo de eventos de un caso de uso puede hacer que el comportamiento "normal" sea más difícil de ver. Extraerlo debería mejorar la comprensibilidad del modelo de caso de uso.

      Asegúrese de que el flujo de eventos del caso de uso base siga siendo completo y comprensible por sí mismo, sin ninguna referencia al caso de uso de la extensión.

      Sólo el caso de uso de la extensión conoce la relación entre los dos casos de uso. El caso de uso base solo sabe que tiene puntos de extensión, no sabe qué casos de uso de extensión los utilizan.

      Describe brevemente cada relación extendida que definas. Definir las condiciones que deben cumplirse para que se produzca la prórroga. Asegúrese de definir el punto de extensión en el caso de uso base en el que se debe insertar la extensión.

      Si no define ninguna condición, significa que la extensión siempre se realiza.
      Si el caso de uso de extensión tiene varios segmentos de comportamiento que se insertarán en diferentes puntos de extensión en el caso de uso base, asegúrese de definir estos segmentos y el punto de extensión para cada segmento en el caso de uso base.

      Establecer generalizaciones entre casos de uso

      Si dos o más casos de uso tienen similitudes en estructura y comportamiento, puede factorizar el comportamiento común para crear un nuevo caso de uso principal . Los casos de uso originales serán entonces casos de uso secundarios en relaciones de generalización con el padre. El caso de uso secundario hereda todo el comportamiento descrito para el caso de uso principal. Consulte también Directrices: modelo de casos de uso y Directrices: generalización de casos de uso .

      Una relación de generalización entre dos casos de uso significa que cuando una instancia de caso de uso sigue la descripción de un caso de uso secundario, también debe seguir la descripción del caso de uso principal para que se considere completa.

      Generalmente, para que valga la pena mantener un caso de uso principal y una relación de generalización con un niño, es necesario que haya al menos dos casos de uso secundarios heredados del mismo padre. Una excepción es si tiene dos casos de uso en los que uno es una especialización del otro, pero ambos deben poder instanciarse de forma independiente.

      Sólo el caso de uso secundario conoce la relación entre los dos casos de uso; ningún caso de uso principal sabe qué casos de uso secundarios lo especializan.

      Para ayudar a otros a comprender el modelo, debe describir brevemente la relación de generalización. Explica por qué creaste la relación de generalización.

      En el flujo de eventos del caso de uso del niño, es necesario explicar cómo el niño modificará las secuencias de comportamiento heredadas insertando nuevos segmentos de comportamiento.

      Establecer generalizaciones entre actores

      Los actores tendrán características comunes que debes modelar mediante generalizaciones de actores. Esta parte del trabajo se realiza mejor después de haber realizado los primeros intentos con un modelo de caso de uso.

      Escriba una breve descripción de las generalizaciones de actores e inclúyalas en los diagramas de casos de uso para obtener más aclaraciones.

      Evalúe sus resultados

      Debe discutir continuamente la incorporación de relaciones de inclusión, extensión y generalización con el cliente y los usuarios, y asegurarse de que tengan una comprensión clara de los casos de uso y actores resultantes, y que estén de acuerdo con sus descripciones.

      Verifique el modelo de caso de uso en esta etapa para verificar que su trabajo va por buen camino, pero no revise el modelo en detalle. Debe revisar y discutir los casos de uso y las relaciones recién incorporados con el cliente y los usuarios para que comprendan claramente los casos de uso y estén de acuerdo con sus descripciones.

      Si es necesario, puede decidir organizar los casos de uso en paquetes de casos de uso . Consulte Directrices: Paquete de casos de uso para obtener más información sobre cuándo considerar esta opción.

      También debe considerar los puntos de control para el modelo de casos de uso mientras trabaja en él. Consulte especialmente los puntos de control para actor, caso de uso y modelo de caso de uso en Actividad: Revisar requisitos.
