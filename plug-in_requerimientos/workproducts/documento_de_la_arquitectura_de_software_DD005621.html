Artifact: Documento de la arquitectura de software
El Documento de arquitectura de software proporciona una descripción general arquitectónica completa del sistema, utilizando varias vistas arquitectónicas diferentes para representar diferentes aspectos del sistema.
Available States: vista de caso de uso
Purpose
El documento de arquitectura de software proporciona una descripción general completa de la arquitectura del sistema de software. Sirve como medio de comunicación entre el arquitecto de software y otros miembros del equipo del proyecto con respecto a decisiones arquitectónicamente importantes que se han tomado en el proyecto.

Description
Main Description

Momento
La representación y los objetivos de la arquitectura del software suelen ser algo que debe definirse antes de las primeras iteraciones y luego mantenerse durante todo el proyecto. Estas pautas de representación arquitectónica están documentadas en las versiones iniciales del Documento de arquitectura de software .

El Documento de Arquitectura de Software se desarrolla principalmente durante la fase de elaboración, porque uno de los propósitos de esta fase es establecer una base arquitectónica sólida.

Es probable que la vista de casos de uso dentro del documento se considere antes que las otras vistas, porque los casos de uso impulsan el desarrollo y son un aporte esencial para la planificación de iteraciones. Para sistemas con un alto grado de concurrencia y distribución, es probable que las vistas de proceso e implementación también se consideren tempranamente, porque entonces podrían tener un impacto sustancial en todo el sistema.

Responsabilidad
Un arquitecto de software es responsable de producir el Documento de arquitectura de software , que captura las decisiones de diseño más importantes en múltiples vistas arquitectónicas.

El arquitecto de software establece la estructura general de cada vista arquitectónica: la descomposición de la vista, la agrupación de elementos y las interfaces entre estas agrupaciones principales. Por lo tanto, en contraste con otros roles, la visión del arquitecto de software es de amplitud , en lugar de profundidad .

El arquitecto de software también es responsable de mantener la integridad arquitectónica del sistema a través del proceso de desarrollo mediante:

Aprobar todos los cambios en elementos arquitectónicamente significativos, como las interfaces principales, descritos en el Documento de arquitectura de software .
Ser parte de la "junta de control de cambios" de decisiones para resolver problemas que impactan la arquitectura del software.
Sastrería
Debe ajustar el esquema del Documento de arquitectura de software para adaptarlo a la naturaleza de su software:

Algunas de las vistas arquitectónicas pueden ser irrelevantes:
La Vista de implementación no es necesaria para sistemas de CPU única.
La Vista de proceso no es necesaria si el sistema utiliza un solo hilo de control.
La vista de datos no es necesaria a menos que la persistencia de objetos sea un aspecto importante del sistema y el mecanismo de persistencia requiera un mapeo entre objetos persistentes y no persistentes.
Algunos aspectos específicos del software pueden requerir su propia sección; por ejemplo, aspectos relacionados con la gestión de datos o cuestiones de usabilidad.
Es posible que necesite apéndices adicionales para explicar ciertos aspectos, como la justificación de ciertas elecciones críticas junto con las soluciones que se han eliminado, o para definir acrónimos o abreviaturas, o presentar principios generales de diseño.
El orden de las distintas secciones puede variar, dependiendo de los actores del sistema y su enfoque o interés.
Las ventajas y desventajas de cada vista arquitectónica son las siguientes:

Vista de caso de uso
Esta vista es obligatoria.

Vista lógica
Esta vista es obligatoria.

Vista de proceso
Esta vista es opcional. Utilice esta vista sólo si el sistema tiene más de un subproceso de control y los subprocesos separados interactúan o dependen unos de otros.

Vista de implementación
Esta vista es opcional. Utilice esta vista solo si el sistema está distribuido en más de un nodo. Incluso en estos casos, utilice únicamente la vista de implementación cuando la distribución tenga implicaciones arquitectónicas. Por ejemplo, en los casos en los que hay un único servidor y muchos clientes, una vista de implementación solo necesita delinear las responsabilidades del servidor y los clientes como una clase de nodos; no es necesario mostrar todos los nodos del cliente si todos tienen las mismas capacidades.

Vista de implementación
Esta vista es opcional. Utilice esta vista sólo en los casos en los que la implementación no se base estrictamente en el diseño, es decir, cuando exista una distribución diferente de responsabilidades entre los paquetes correspondientes en los modelos de Diseño e Implementación. Si el empaquetado de los modelos de diseño e implementación es idéntico, esta vista se puede omitir.

Vista de datos
Esta vista es opcional. Utilice esta vista sólo si la persistencia es un aspecto importante del sistema. y la traducción del modelo de diseño al modelo de datos no se realiza automáticamente mediante el mecanismo de persistencia.
